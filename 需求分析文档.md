# 青年同城互助平台 — 需求分析文档

> 版本：v1.1  
> 日期：2026-05-25  
> 状态：修订版（简化版，适配大学实训项目）

---

## 一、项目概述

### 1.1 项目背景

当前城市中大量青年处于独居状态，面临社交圈窄、生活成本高、技能提升渠道有限等问题。传统的社交平台偏重娱乐属性，缺乏实际的互助机制；二手交易平台仅解决物品流通，无法建立持续性信任关系。青年群体亟需一个以"互助"为核心、以"同城"为边界的综合性社区平台，将社交、技能、资源进行有机整合。

### 1.2 项目定位

**青年同城互助平台**是一个面向城市单身青年群体的互助型社区产品，核心价值在于：通过技能交换、物品共享、临时搭伴三大功能模块，配合信用评分体系和同城推荐，构建一个"以互助促社交、以信用建信任"的良性生态。

### 1.3 项目目标

| 维度 | 目标 |
|------|------|
| 用户价值 | 降低生活成本、拓展社交圈、提升个人技能 |
| 商业价值 | 积累同城用户流量，后续可通过增值服务、广告、企业合作变现 |
| 社会价值 | 促进资源循环利用、减少城市青年孤独感、构建互助文化 |

### 1.4 项目性质

> **本项目为大学实训项目**，以学习实践和功能完整性为主要目标，不追求商业级高可用。部分复杂功能（支付、微信登录、实名认证、短信服务等）暂不实现，线下自行结算。地图服务使用百度地图。

## 二、目标用户分析

### 2.1 用户画像

| 维度 | 描述 |
|------|------|
| 年龄段 | 20-35岁 |
| 身份 | 在校大学生、职场新人、城市漂泊青年 |
| 状态 | 单身、独居或合租 |
| 特征 | 有技能但社交圈有限、有闲置资源但利用率低、有社交需求但渠道少 |
| 痛点 | 生活成本高、技能提升贵、社交困难、闲置物品浪费 |

### 2.2 典型用户角色

**角色一：技能型用户 — 小张**
- 25岁程序员，擅长编程和英语，想学吉他和健身
- 希望找到能互相教学的同城伙伴

**角色二：资源型用户 — 小李**
- 23岁应届毕业生，刚租房缺少生活用品
- 有闲置书籍和数码配件，希望交换或共享

**角色三：社交型用户 — 小王**
- 28岁设计师，周末经常独自一人
- 希望找人拼饭、看展、短途旅行

### 2.3 用户核心需求

1. **技能互补**：找到技能匹配的交换伙伴
2. **资源共享**：让闲置物品发挥价值
3. **临时搭伴**：解决"一个人"的尴尬
4. **安全信任**：确保互助对象可靠
5. **便捷高效**：同城精准推荐，减少无效沟通

### 3.2 暂不实现的功能

> 以下功能因实训项目条件限制，暂不纳入开发范围：

| 功能 | 原因 |
|------|------|
| 微信授权登录 | 需要微信开放平台认证，申请流程复杂 |
| 短信验证码登录 | 需要短信服务商（如阿里云SMS），有费用和审核要求 |
| 支付功能 | 涉及支付接口接入，安全要求高，实训项目不需要 |
| 实名认证 | 需要对接身份证验证接口，有隐私合规要求 |
| WebSocket 实时推送 | 技术复杂度高，轮询方式可满足基本需求 |
| LBS 精准定位 | 需要精确定位权限，简化为城市选择 |
| 智能推荐算法 | 算法复杂度高，简化为按城市+信用分排序 |

### 3.3 详细功能描述

#### 3.3.1 用户模块

| 功能项 | 优先级 | 描述 |
|--------|--------|------|
| 用户名注册/登录 | P0 | 用户名 + 密码注册登录，简单可靠 |

| 技能标签管理 | P0 | 自定义添加"我能教"和"我想学"的技能标签 |

#### 3.3.2 技能交换模块

| 功能项 | 优先级 | 描述 |
|--------|--------|------|
| 技能发布 | P0 | 选择"我能教"/"我想学"，填写技能名称、描述、可交换时间、偏好地点 |
| 技能列表浏览 | P0 | 按技能分类、信用分排序浏览 |
| 技能匹配推荐 | P0 | 基于用户的"能教/想学"标签自动匹配推荐交换对象 |
| 交换请求 | P0 | 向目标用户发起交换请求，附留言说明 |
| 交换确认 | P0 | 接受/拒绝请求，确认后进入"进行中"状态 |
| 交换进度跟踪 | P1 | 记录每次交换的时间、内容、完成状态 |
| 交换完成确认 | P0 | 双方确认完成后，更新信用分 |

**核心流程：**

```
发布技能 → 系统匹配推荐 → 浏览/搜索 → 发起交换请求
→ 对方确认 → 交换进行中 → 双方确认完成 → 互评 → 信用分更新
```

#### 3.3.3 闲置共享模块

| 功能项 | 优先级 | 描述 |
|--------|--------|------|

| 物品浏览 | P0 | 分类浏览、搜索、同城筛选 |
| 物品申请 | P0 | 提交借用/交换申请，附留言 |
| 物品状态管理 | P0 | 可用/已借出/已送出/已下架 |
| 归还确认 | P0 | 借用物品归还后双方确认 |
| 物品评价 | P0 | 完成后对物品状态和对方信用进行评价 |

#### 3.3.4 临时搭伴模块

| 功能项 | 优先级 | 描述 |
|--------|--------|------|
| 活动发布 | P0 | 选择类型（拼饭/拼展/拼旅行/其他）、时间、地点、人数限制、费用说明、备注 |
| 活动浏览 | P0 | 按类型、时间、同城筛选 |
| 活动报名 | P0 | 提交报名，发起人审核 |
| 活动管理 | P0 | 发起人管理报名列表、确认/拒绝参与者 |
| 活动完成确认 | P0 | 活动结束后双方确认参加 |
| 活动回顾 | P1 | 活动结束后发布照片和感受 |

#### 3.3.5 信用评分模块

| 功能项 | 优先级 | 描述 |
|--------|--------|------|
| 信用分计算 | P0 | 基于互助完成度、评价、违规记录自动计算 |
| 信用等级展示 | P0 | 信用分对应等级徽章（新手/铜牌/银牌/金牌/钻石） |
| 互助记录 | P0 | 展示历史互助记录，包括类型、对象、时间、评价 |
| 信用分影响 | P0 | 信用分影响发布权重、推荐排序 |
| 信用分变动通知 | P1 | 信用分变动时推送通知 |

**信用分计算规则：**

| 行为 | 分值变动 |
|------|----------|
| 完成一次技能交换 | +5 |
| 完成一次物品共享 | +3 |
| 完成一次临时搭伴 | +2 |
| 获得好评 | +1~+3（按星级） |
| 未完成已确认的互助 | -5 |
| 收到差评 | -1~-3 |
| 违规被举报核实 | -10 |

**信用等级划分：**

| 等级 | 分值范围 | 徽章 |
|------|----------|------|
| 新手 | 0-20 | 🌱 |
| 铜牌 | 21-50 | 🥉 |
| 银牌 | 51-100 | 🥈 |
| 金牌 | 101-200 | 🥇 |
| 钻石 | 200+ | 💎 |

#### 3.3.6 同城推荐模块

| 功能项 | 优先级 | 描述 |

## 四、非功能需求

### 4.1 性能需求

| 指标 | 要求 |
|------|------|
| 页面首屏加载时间 | ≤ 3秒 |
| API 响应时间 | ≤ 500ms（95%请求） |
| 并发支持 | 支持100并发用户（实训项目标准） |
| 图片加载 | 懒加载，单图 ≤ 500KB |

### 4.2 安全需求

| 项目 | 要求 |
|------|------|
| 用户认证 | JWT Token + Refresh Token |
| 敏感信息 | 密码 BCrypt 加密存储 |
| 防护措施 | XSS 防护、SQL 注入防护、接口限流 |
| 隐私保护 | 位置信息仅展示到城市级别 |

### 4.3 可用性需求

| 项目 | 要求 |
|------|------|
| 系统可用性 | ≥ 99%（实训项目标准） |
| 浏览器兼容 | Chrome、Firefox、Safari、Edge 最新两个版本 |
| 移动端适配 | 响应式布局，适配主流手机屏幕 |

### 4.4 可扩展性需求

- 数据库设计预留扩展字段
- API 版本化管理（/api/v1/）
- 模块化设计，便于后续迭代

### 5.2 核心用例详述

#### UC-01：技能交换

| 项目 | 内容 |
|------|------|
| 用例名称 | 技能交换 |
| 参与者 | 用户A（技能提供方）、用户B（技能需求方） |
| 前置条件 | 双方均已注册并完善技能标签 |
| 主流程 | 1. 用户A发布"我能教编程"<br>2. 系统匹配"想学编程"的用户列表推荐给A<br>3. 用户B浏览到A的发布，发起交换请求（我能教健身，想学编程）<br>4. 用户A查看B的资料和信用分，确认交换<br>5. 双方线下/线上完成技能交换<br>6. 双方在平台确认完成并互评 |
| 后置条件 | 双方信用分更新，互助记录生成 |
| 异常流程 | 3a. B直接被A拒绝 → 通知B<br>5a. 一方未完成 → 另一方可申诉<br>6a. 超过7天未确认 → 系统自动标记待处理 |

| 项目 | 内容 |
|------|------|
| 用例名称 | 闲置物品共享 |

| 前置条件 | 物品所有者已发布闲置物品 |
| 主流程 | 1. 所有者发布物品（图片+描述+交换方式）<br>2. 需求者浏览物品，提交借用/交换申请<br>3. 所有者审核申请，确认借出<br>4. 双方线下交接<br>5. 借用到期后归还，或赠送/交换完成<br>6. 双方确认完成并评价 |
| 后置条件 | 物品状态更新，信用分更新 |

#### UC-03：临时搭伴

| 项目 | 内容 |
|------|------|
| 用例名称 | 临时搭伴 |
| 参与者 | 活动发起人、参与人 |
| 前置条件 | 发起人已发布搭伴活动 |
| 主流程 | 1. 发起人发布搭伴活动（类型+时间+地点+人数）<br>2. 参与人浏览活动，提交报名<br>3. 发起人审核报名，确认参加人<br>4. 活动当天线下集合<br>5. 活动结束后双方确认参加并评价 |

## 七、约束与假设

### 7.1 约束

1. **实训项目**：以学习实践为主，不要求商业级高可用
2. **团队规模**：假设3-5人小团队开发
3. **开发周期**：8-10周
4. **预算**：零预算为主，使用开源技术栈
5. **合规**：涉及用户信息需符合《个人信息保护法》

### 7.2 假设

1. 用户主要通过PC浏览器访问
2. 初期以一个城市为试点
3. 信用分体系采用简化模型，后续迭代优化
4. 暂不需要支付功能，线下自行结算
5. 登录采用用户名+密码方式，降低接入门槛
