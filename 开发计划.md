## 一、技术选型

### 1.1 前端技术栈

| 技术 | 选型 | 理由 |
|------|------|------|
| 框架 | **Vue 3** + Composition API | 生态成熟、上手快、适合中小型项目 |
| 构建工具 | **Vite** | 开发热更新快、配置简单 |
| UI组件库 | **Element Plus** | 中文文档完善、组件丰富、社区活跃 |
| 状态管理 | **Pinia** | Vue 3 官方推荐、轻量灵活 |
| 路由 | **Vue Router 4** | Vue 3 标配 |
| HTTP 请求 | **Axios** | 拦截器机制完善、广泛使用 |
| 地图 | **百度地图 JS API** | 国内定位精度高、免费额度够用、文档完善 |
| 移动端适配 | **postcss-pxtorem** + **amfe-flexible** | 自动适配不同屏幕尺寸 |

### 1.2 后端技术栈

| 技术 | 选型 | 理由 |
|------|------|------|
| 框架 | **Spring Boot 3** | Java 生态主流、企业级成熟、实训项目主流选择 |
| 安全框架 | **Spring Security** + **JWT** | 认证授权标准方案 |
| ORM | **MyBatis-Plus** | 简化 CRUD、代码生成器、分页插件 |
| 数据库 | **MySQL 8.0** | 关系型数据适合、免费、社区成熟 |
| 缓存 | **Redis** | Token 存储、热点数据缓存 |
| 对象存储 | **MinIO** | 图片/文件存储、自部署无费用 |
| 接口文档 | **Knife4j** (Swagger 增强) | 接口文档自动生成、在线调试 |
| 日志 | **Logback** + **SLF4J** | Java 标准日志方案 |

> **注意**：相比初版，移除了 RabbitMQ（消息队列），消息通知使用简单的轮询方式，降低系统复杂度。

### 1.3 DevOps

| 技术 | 选型 | 理由 |
|------|------|------|
| 版本管理 | **Git** + **GitHub/Gitee** | 团队协作标准 |
| 容器化 | **Docker** | 环境一致性、部署便捷 |
| 反向代理 | **Nginx** | 静态资源托管、API 转发 |

t_review (id, record_id, reviewer_id, reviewee_id, rating, content, created_at)

t_message (id, sender_id, receiver_id, content, type, is_read, created_at)

-- 系统通知表
t_notification (id, user_id, title, content, type, is_read, created_at)

## 六、开发规范

### 6.1 Git 分支策略

```
main          ─── 生产分支，稳定代码
  └── develop ─── 开发分支，日常集成
       ├── feature/user-module      ─── 功能分支
       ├── feature/skill-exchange
       ├── feature/goods-share
       ├── feature/activity
       ├── feature/credit-system
       └── hotfix/xxx               ─── 紧急修复
```

- 功能分支从 `develop` 拉出，完成后提 PR 合并回 `develop`
- `develop` 测试通过后合并到 `main`
- 提交信息格式：`[模块] 动作: 描述`，如 `[用户] feat: 新增用户名密码登录接口`

### 6.2 接口规范

- RESTful 风格：`/api/v1/{模块}/{资源}`
- 统一响应格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

- 分页统一参数：`pageNum`、`pageSize`
- 认证方式：`Authorization: Bearer {token}`

### 6.3 代码规范

- 后端：阿里巴巴 Java 开发手册
- 前端：ESLint + Prettier（Vue 官方风格指南）
- 数据库：表名 `t_` 前缀，字段名蛇形命名，索引规范命名
- 注释：公共方法必须添加注释，复杂业务逻辑添加行内注释

## 九、风险与应对

| 风险 | 应对措施 |
|------|----------|
| 技术难点超预期 | 优先保证核心流程跑通，P2 功能可延期 |
| 团队成员进度不一致 | 每周站会同步进度，提前暴露风险 |
